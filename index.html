The question: how much code do you need for an always-on personal AI agent?

Claude Code shipped with a crab mascot. The community ran with it. Peter Steinberger built OpenClaw — 400K lines connecting every messaging app. NanoClaw by Gavriel Cohen asked: what if we did it in 500 lines? The crustacean theme is fun but the real question is architectural.

OpenClaw vs NanoClaw

OpenClaw

  • 🔴 ~400,000 lines / 70+ deps
  • 🔴 Application-level sandboxing
  • 🔴 Shared Node.js process
  • 🔴 CVE-2026-25253 (CVSS 8.8)
  • 🔴 12% of ClawHub skills malicious
vs

NanoClaw

  • ✅ ~500 lines core / 6 deps
  • ✅ OS-level container isolation
  • ✅ Separate Linux VM per group
  • ✅ Auditable in 8 minutes
  • ✅ Skills = .md files (no registry)

NanoClaw philosophy: "Don't add features. Add skills." — Messaging channels (Telegram, Slack, Discord) are added via /add-telegram skills that transform your fork, keeping the core tiny.

Side-by-side. OpenClaw has 400K lines in a shared process — one compromise and everything is exposed. NanoClaw uses real container isolation. The skills philosophy is key: instead of building in every messaging adapter, you add them via Claude Code skills. /add-telegram transforms your fork. The core stays small and auditable.

Harness Engineering

2025 was agents. 2026 is agent harnesses. — source

What is a harness?

The system wrapping the AI model for reliable long-running tasks. (Anthropic blog)

  • Permissions & sandboxing
  • State persistence across sessions
  • Tool orchestration (MCP servers)
  • Lifecycle (launchd, containers)
  • Context engineering — right context, right tools, right time
"I haven't written a single line of code myself since December 2025. I run 10-20 AI agents in parallel." — Andrej Karpathy, March 2026

The pattern: keep the core tiny, make it extensible via skills, iterate — add context, tools, and autonomy gradually.

Karpathy's AutoResearch: 630 lines. NanoClaw: ~500 lines. Same insight.

Anthropic published guidance on effective harnesses for long-running agents. The harness manages permissions, state, tool orchestration, lifecycle. Karpathy built AutoResearch in 630 lines. NanoClaw: 500 lines. The key insight: you don't build a big app. You build a tiny harness and iteratively add skills, context, and tools. Context engineering — giving the agent the right information at the right time — is the real skill.

NanoClaw Architecture

Walk through the diagram. Host process handles Telegram messaging and routing. Each group gets its own container — a real Linux VM. Containers are EPHEMERAL: spawned per message, destroyed after with --rm. No persistent state inside the container. This is key: you only pay compute when something happens. NanoClaw recently partnered with Docker for cross-platform support — Apple Container on macOS, Docker on Linux.

container-runner.ts — mount isolation

if (isMain) {
  mounts.push({ hostPath: projectRoot,
    containerPath: '/workspace/project' });
} else {
  // Non-main: ONLY their own folder
  mounts.push({ hostPath: groupDir,
    containerPath: '/workspace/group' });
}
// Per-group isolated sessions
mounts.push({ hostPath: groupSessionsDir,
  containerPath: '/home/node/.claude' });

// Only 2 env vars exposed to containers
const allowedVars = [
  'CLAUDE_CODE_OAUTH_TOKEN',
  'ANTHROPIC_API_KEY'
];
Security comparison. OpenClaw does application-level sandboxing in a shared process — if one adapter is compromised, everything is exposed. NanoClaw uses OS-level containers. Each group is a separate Linux VM. The mount allowlist lives outside the project directory at ~/.config/nanoclaw/ — containers literally cannot modify their own security config. Only 2 environment variables are exposed. This is the actual code from container-runner.ts.

Ephemeral Containers

Containers spawn only when needed — on message or cron trigger. Fresh environment each time (--rm). No idle resource consumption. State persists via mounted volumes, not in the container.

Each addition makes the agent more capable. You don't build everything at once. You iterate. Frequency deviation → agent checks load balance, weather forecast, demand schedule → adjusts demand response OR escalates with recommendation.

The shift: humans made heuristic decisions ("if pressure drops 10%, shut down"). Agents do contextual reasoning — checking history, correlations, runbooks — and escalate when uncertain.

Here's where personal meets professional. The same event-driven pattern maps directly to industrial applications. Pipeline monitoring: a pressure sensor event triggers an agent that loads historical data, analyzes the anomaly, and either isolates a valve autonomously or pages an engineer with full context. And critically: they escalate when uncertain rather than making a bad autonomous decision.
